About TigerMUD

TigerMUD is an open source Multi User Dungeon (MUD) for Microsoft Windows, written in C#.

You can learn more at http://tigermud.com

Who should read this guide

This Developer’s Guide is for those who want to configure the Tortoise CVS source control client, download the source code from SourceForge, compile your own executables, and/or changes the database to MySQL or Microsoft SQL.

To avoid this work, download and run the TigerMUD Windows install package (MSI file) that installs and configures everything for you, including the internal Access database to get running fast. Using this method, you don’t get the bleeding edge code, but you get the latest declared “stable” build.

You can get both the TigerMUD install program and this Developer’s Guide under the “Latest File Releases” section at:

http://sourceforge.net/projects/tigermud/

Report bugs and ask questions in the discussion forum at the TigerMUD project Web page:

http://sourceforge.net/projects/tigermud/

Default TigerMUD installation on Windows with MS Access (PREFERRED METHOD)

You can perform the quickest and easiest TigerMUD installation if you simply run the TigerMUD_Setup.msi file. This creates a default installation of TigerMUD on your local computer and uses MS Access as its back-end database. All options are preconfigured for you.

If you wish to then perform development tasks based on this default installation, follow the steps in this document but skip those steps for MySQL software and components, and the database import steps. The sample Access database available in the source code requires no import steps to use it.

Choosing your database type is done by editing the tigermud.xml file and setting the <Database><Type> parameter to msaccess, mysql, or mssql.

TigerMUD installation and configuration on Windows with MySQL

(These steps were tested on November 8, 2004 on Windows XP. The product versions are listed to provide a known-good reference platform. Other versions may work, too.)

TortoiseCVS Install and Config

Download and install TortoiseCVS 1.8.12 from:

http://www.tortoisecvs.org/download.shtml

Create a c:\TigerMUD folder (or wherever you want), right-click the new folder, and then click CVS Checkout...

The TortoiseCVS Checkout Module dialog appears.

Here, you must choose between the next two procedures:

1.   Setting up anonymous read-only source code access.

2.   TigerMUD dev team setup for read/write source code access.

For anonymous (read-only) source code access:

For CVSROOT string, paste in:

-d :pserver:anonymous@cvs.sourceforge.net:/cvsroot/tigermud

 

This action will populate other fields accordingly. Set your Module=TigerMUD (case sensitive). Click OK, then all the latest code will download into your folder.

For TigerMUD developer team members (read/write access):

For CVSROOT string, paste in:

:ssh:username@cvs.sourceforge.net:/cvsroot/tigermud

where <username> is replaced with your Sourceforge account username. This action will populate other fields accordingly.

Module=TigerMUD (case sensitive)

Click OK, then you will be prompted for your Sourceforge password, then all the latest code will download into your folder.

MySQL Install and Config

(Skip these steps if using the Default Installation w/Microsoft Access.)

Download the following packages from here:

http://dev.mysql.com/downloads/

1.      MySQL 4.1.7 server (Windows Essentials (x86))

2.      MySQL Connector/ODBC 3.5

3.      MySQL Administrator 1.0.14

4.      MySQL Query Browser 1.1.1

 

Note that the tigermud.xml file controls the database type. For MySQL, your database section should look like this:

<Database>

<Type>mysql</Type>

Install MySQL 4.1.7 (Windows Essentials (x86))

1.      Choose Custom setup

2.      Choose MySQL Server and Client Programs

3.      Check “Configure MySQL Now”

MySQL Server Instance Configuration Wizard appears.

1.      Choose Detailed Configuration.

2.      Choose Server Machine.

3.      Choose Non-Transactional Database Only.

4.      Choose Manual Setting - set concurrent connections to 500.

5.      Leave default value of port 3306.

6.      Choose Standard Character Set

7.      Choose Install As Windows Service - Service Name=MySQL - check Launch MySQL Server automatically

8.      Choose Modify Security Settings - pick a root password and then confirm –

9.      Check Root may only Connect from localhost.

10.  Ensure you get a success message after the setup program completes.

Install MySQL Connector/ODBC 3.5

No prompts to choose from. Just use the defaults.

Install MySQL Administrator 1.0.14

Choose Complete Setup.

Install MySQL Query Browser 1.1.1

Choose Complete Setup.

Run MySQL Administrator

·        username=root

·        password=<the root password you chose above>

·        hostname=localhost

·        port=3306

 

In the left pane click Restore. On right pane, click Open Backup File. Navigate down to mudmysql_import.sql and choose it. Check Create database(s) if they don’t exist.

·        Target Schema=Original Schema

·        Backup type=SQL files

·        File Charset=utf8

 

Click Start Restore. Ensure that you get a success message.

Run MySQL Query Browser

The Connection information is the same as MySQL Administrator.

·        Schema=tigermud

In the right pane, double-click one of the tables under the tigermud schema and click Ctrl-Enter.

You should get a view of all the data in that table. You can edit this data by clicking the Edit button at the bottom of the screen. This lets you modify cells, then click Apply Changes when you’re done.

Modify your TigerMUD.xml file

Open the TigerMUD.xml file in the tigermud source folder and specify the database servername, database username (root in this example), and the password for that user account. For better security, create a separate user for the TigerMUD database and use that. Save the xml file.

Use SharpDevelop or Microsoft Visual Studio.NET?

Here, you must choose between writing code with Microsoft Visual Studio.NET or the free open source version named SharpDevelop.

Using SharpDevelop

Download SharpDevelop 1.0.1.1649 from here:

http://www.icsharpcode.net/OpenSource/SD/Download/

Newer versions of SharpDevelop might work for you, too. Run the setup package. There are no special options to select, so just use the default setup values.

 

  1. Choose the “Create new code completion database” option when you run SharpDevelop the first time.
  2. Choose Use SharpDevelop application directory.
  3. Click Start database creation. It will take a minute to create the database.
  4. When SharpDevelop opens, dismiss the Tip of the Day and click ‘Import Project’.
  5. Choose the TigerMUD.sln file as the project that you want to import. SharpDevelop will do some conversions on it to let you do development with it.

 

The last step is that you must add the steps in the vspostbuild.bat file to your SharpDevelop combine project options. These are steps that occur upon completion of a compile. In Visual Studio.NET, developers have to run this manually, but SharpDevelop lets you add steps to run at compile completion.

Using Microsoft Visual Studio .NET

Install Microsoft Visual Studio.NET and ensure that Visual C# is chosen. Once install is complete, then double-click the TigerMUD.sln file and the TigerMUD development project will open.

Important Note for both SharpDevelop and Visual Studio.NET users

Ensure you have the Microsoft .NET Framework version 1.1 installed, since TigerMUD uses the ODBC .NET Data Provider and it is included in the .NET Framework 1.1 package. TigerMUD is not tested or supported on .NET Framework 1.0.

Get Microsoft .NET Framework 1.1 here:

http://msdn.microsoft.com/netframework/downloads/framework1_1/

Compile and Run TigerMUD with SharpDevelop

In the SharpDevelop window with the TigerMUD combine file open, click Build, then click Rebuild Combine (Alt-F8). Then run the MUD by clicking Debug, then click Run (F5). This compiles and runs TigerMUD. Watch the server console to ensure TigerMUD successfully finds the tigermud.xml file and Connects to the database. If all succeeds, then tigermud will indicate that it is waiting for connections.

Compile and Run TigerMUD with Visual Studio.NET

With the TigerMUD project open, click Build->Build Solution (Ctrl-Shift-B). IMPORTANT: Before clicking Start, make sure you run the vspostbuild.bat file in your TigerMUD folder. Run this batch file after every compile before you start the server. The batch file solves file locking problem we encountered with Visual Studio.NET.

You can accept all the defaults from the TigerMUD install program and only have to edit the tigermud.xml file to specify your SMTP server for new player account activation. There is also a setting in the tigermud.xml file to turn this feature off if you do not have access to an SMTP server, or simply do not want to use this feature. Once you configure the tigermud.xml file, you can run the server using the desktop icon provided.

Using the Nant Script

TigerMUD can be built using Nant 0.85 (rc3). You can download Nant here:

http://nant.sourceforge.net/.

Setting up Nant for your environment

Download Nant and unzip it to a directory on your computer. The Nant executable resides in <nant root directory>\bin. Add this path to your Path environment variable. On Windows, you will add this by:

1.      Right click on My Computer

2.      Select Properties

3.      Click on the Advanced tab

4.      Click on Environment Variables

5.      In System Variables, look for the PATH variable

6.      Click Edit

7.      Add your path to the END of the string, make sure you have a semicolon “;” separating your entry from the previous one

8.      Click OK twice

 You are now ready to use Nant.

Building TigerMUD with Nant (on Windows, using Microsoft’s .Net Framework 1.1)

Open up a command prompt and change to the TigerMUD root directory. Type the following to start the build:

nant

The Nant executable will detect the “tigermud.build” file and start building TigerMUD using Microsoft’s .Net compiler. The output will be stored in:

<TigerMUD Root Directory>\build\net-1.1\

It will be ready to run. Simply change to this directory and run TigerMUD.exe.

Building TigerMUD with Nant and MONO (1.1.7)

This step requires you to have Mono 1.1.7 or greater installed on your computer. You can download

Mono from here:

http://www.mono-project.com/Downloads

To compile TigerMUD with Mono, you will change to the TigerMUD root directory and type the following:

nant -t:mono-1.0

The -t flag tells Nant to use the Mono compiler (msc) instead of the .Net compiler.

The output will be stored in:

<TigerMUD Root Directory>\build\mono-1.0\

Running TigerMUD on MONO

This step requires you to have Mono 1.1.7 or greater installed on your computer. You can download

Mono from here:

http://www.mono-project.com/Downloads

For Windows users: once you have installed Mono on your machine, you will need to open the Mono command prompt, so that the correct path settings are in place. You can access it by going Start -> Programs -> Mono 1.1.7 For Windows -> Mono 1.1.7 Command Prompt.

Change to:

<TigerMUD Root Directory>\bin\debug

and run the following command:

mono tigermud.exe

You will see the TigerMUD engine starting up.

For Nant Users

If you have compiled TigerMUD using Nant, using either Mono’s compiler or Microsoft’s compiler, then change to the appropriate directory:

Either <TigerMUD Root Directory>\build\mono-1.0 or <TigerMUD Root Directory>\build\net-1.1\

In this directory, you will run the following command:

mono tigermud.exe

Running TigerMUD for the first time

Any time you run TigerMUD through either the desktop icon or from Visual Studio/SharpDevelop, TigerMUD displays all its startup values on the server console, so if you encounter errors, check those values first to ensure they are what you require in your environment.

Connect to TigerMUD

The default TigerMUD port is 8000 (configured with the tigermud.xml file). To connect, click Start, Run, and type ‘telnet localhost 8000’. Check the “mudactors” database table for a few valid usernames and passwords that are already configured for you.

Troubleshooting

If something goes wrong running TigerMUD, the server console will display exceptions and error messages that you can use to troubleshoot. We are always on the lookout for bugs and new features and would greatly appreciate your feedback. Go to our SourceForge project page and click on either Bugs or RFE (Request for Enhancement) to log your bug or feature request.

Contact Us

If you questions/comments, please post in the public forum at the TigerMUD project Web page. Also, there are links available to file bugs and feature requests at this Web page:

http://sourceforge.net/projects/tigermud/

TigerMUD Developer Guidelines

These are the rules that the TigerMUD developer team follows. If you are not a member of the TigerMUD team, then feel free to use these guidelines if they are useful to you in your own project.

If you’ve only coded by yourself, then coding with a team takes some getting used to. We’re happy to help ease the transition however we can, just ask!

·     First rule which is most important: DO NOT add other people’s code to TigerMUD. We only have rights to use your code, not something you copied from the Internet. However, if we need some functionality and there is GPL licensed code out there for it, then I may approve making exceptions, but only when we can get clear rights to use the code from the author.

·    If you have an idea, please scan through the forums to see if we’ve already discussed it. If we haven’t, then bring it up in the forums.

·    Check out the info and diagram on tigermud.com and absorb the overall architecture, then scan through the code to see how the command processor works, review the database tables, and User, Item, Mob, and Room classes.

·    Keep an eye on the RFE link on the SourceForge site and look at the features we’re currently interested in. Come up with an implementation of it (pick a simple one at first please).

·    Always record your code changes in TigerMUD_Changes.txt.

·    Comment your code liberally so others can understand your code.

·    Discuss any major TigerMUD architectural changes in the forums first.

·    Do your best to minimize database schema changes (this will be easier the farther along we get in the new database code). Changing the schema requires all the other developers to reload the database, so please only do it when it counts.

·    Usability and readability ranks above performance concerns, unless the performance issues are major. We can always re-factor code for performance later, but it’s really hard for anyone to do that if you don’t focus on usability and readability first. ;-)

·    Compile your code and do not commit to CVS if you have errors. It causes BIG headaches for other developers if you commit broken code and can bring slow development. So, if you change existing code, compile and test it carefully first.

·    Check dependencies on code you change. Will it break someone else’s code?

·    Class names begin in uppercase.

·    Public class properties and methods begin in uppercase.

·    Objects instantiated from the classes begin in lowercase.

·    Use names like ‘tmpitem’ and ‘chkitem’ for temporary objects in loops and such.

·    Use label counters with useful names where possible, rather than just counter1, counter2, etc. instead use things like roomcounter, itemsinbag, characternumber, etc.

·    Do not assume the operating system in your code. We strive to support TigerMUD using Mono on Linux in the future, so beware any OS-specific code. Most of the things offered in the .Net Framework are pretty safe. Ask in the forum if you’re not sure.

·    No stored procedures in SQL. That makes it database specific, we don’t want that.

·    Do not write code that has software dependencies outside the stated TigerMUD platform. For example, don’t write code that makes calls to software or DLLs that are not in the install guide.

·    Add sparingly to Library.cs by sticking to OO programming practices. For example, if you find yourself writing a routine that only takes a user instance as a parameter, should that routine become a method of the User class itself? Think about the objects that your code interacts with, and match it up with the objects themselves when possible.

·    Strive for code reuse. If you are going to use a block of code more than once relatively unchanged, then make that code a method of an existing class, or put it in Library.cs. Sometime you have to spend extra time tweaking your code blocks to make them usable more than once, but it’s usually worth it.

·    Check the Library.cs and the existing classes to make sure you don’t try to duplicate code that has been already written. Pre-tested and debugged code is like gold when faced with writing something from scratch.

 

Happy Mudding!
